    01Introduction: ATEC2026 B2w 任务的双重张力

    ATEC2026 B2wPiper 平台对控制策略提出了两条彼此独立但又必须共存的要求。一方面，从 Task A 地形穿越到 Task B 移动抓取、Task D 越障改造，均要求策略能在长时间内提供稳定速度跟踪——即保持对线速度和角速度指令的精确跟随——同时将能耗降至最低。这一需求承袭自文献 [2403.20001] 所倡导的能耗驱动步态优化哲学，它本质上是一种单目标导向的连续控制问题，奖励函数以速度误差与能量消耗的加权组合为核心。另一方面，同一平台必须容纳多种异质技能——walk、wheel-roll、climb、recover——并在单一策略中实现无缝切换。这恰恰是 MUJICA 架构所要解决的核心挑战。MUJICA 为每种技能分配了一个唯一的显式指示符 ζ 作为观测输入，以此分离不同任务的观测空间，从而在共享网络后端实现多技能联合训练；其真实实验已验证，机器人可在连续任务中自动切换技能（如从跌倒恢复转为爬升 1-m 平台，Fig. 1 顶行与底行）。然而，这种显式指示符机制要求设计者预先定义 skill 切换的规则，本质上是一个高层分类器加底层策略的层级结构。这引出了本模板要回答的核心问题：能否不引入显式技能指示符，让强化学习策略既保持能耗驱动的步态涌现（即长时间稳定速度跟踪 + 能耗最小化），又拥有 MUJICA 级别的多技能切换能力？我们给出的答案是“能”。其关键在于将传统的 ζ 从观测空间迁移到奖励函数分支（reward branch）的隐式表示中：不再由高层 selector 显式指定当前任务，而是利用 MUJICA 中已有的状态估计器（state estimator）所推断的运动状态（如基座线速度估计 ûₜ、轮地距离估计 v̂ₜ）自动驱动模式选择。这样，策略在训练过程中通过 reward branch 隐式地学习何时切换行为，而无需在观测中保留一个离散的 skill tag。MUJICA 的消融实验（Fig. 4）已表明，缺少 velocity 或 wheel-height 估计会严重损害非恢复类任务的性能，这恰恰证明了这类隐式状态对于区分任务不可替代。因此，将 ζ 隐式化不仅可行，而且有望统一能耗优化与多技能集成——这正是本文后续方法部分要详细展开的设计。

    02Two Reward Architectures Confronted: Energy-Centric vs. Skill-Selector

    在奖励-策略接口的设计上，2403.20001 与 2605.13058 代表了两种根本对立的哲学。前者主张「一个能耗项替代所有启发式」，将策略构造为 R = (R_motion + α_en R_en) · exp(−R_aux)，其中 α_en ≈ 1.0 是跨速度域统一的超参数，步态作为能量优化的涌现结果出现，不依赖任何显式技能标识。后者（MUJICA 所属的框架）则主张「一个策略学多种技能」，把技能指示变量 ζ_t ∈ {1,2,3} 直接塞入 53 维 proprio 观测，训练时给每个技能分配独立的 reward branch 和 constraint critic（基于 P3O 约束优化），从而让策略针对不同 ζ_t 学到截然不同的步态模式。

    冲突的焦点在于：MUJICA 的 ζ_t 相当于给策略一个显式 hint，当 ζ=climb 时策略自然切换到高平台攀爬对应的动作空间，这与 2403.20001 所追求的「让步态从能耗自动涌现」直接对立——后者的假设是只要能量项足够好，最优步态会在优化过程中自发形成，无需预设技能边界。然而，消融实验（TABLE III）表明，取消 ζ_t 后统一奖励训练往往崩溃（shared reward）或成功率大幅下降（separate reward 在高平台攀爬上仅 38%），说明纯能量驱动在轮腿机器人多技能场景下难以收敛到可用的策略分化。

    本章提出折中方案：保留 MUJICA 已有的 state estimator（S1）与 reward 分支结构，但 ζ_t 不再进入观测向量；取而代之，用状态估计器输出的 û_t（轮-地距离）与 v̂_t（基线速度）构造联合判别函数，动态地为各 reward branch 分配加权系数。这样既保留了技能特异化的 reward 结构（每个技能仍可根据自身目标设计跟踪奖励与约束），又从观测中删除了显式类别标签，迫使策略从连续状态变量中自主判别当前任务需求。折中方案的核心假设是：轮-地距离与基线速度的联合特征空间足以区分全向移动、高平台攀爬和跌倒恢复三种技能——当 û_t 接近平台高度且 v̂_t 指向平台时，加权应倾向于攀爬分支；当 û_t 很小且 v̂_t 为零时，加权向跌倒恢复分支偏斜——从而在「显式指令」与「纯能量涌现」之间找到一条工程上可行的中间路径。

    03Skill Indicator ζ as Inferred State (Core Design Decision)

    在 MUJICA 原设计中（方案 α），skill indicator ζ 作为显式观测输入直接拼接到低层策略的状态空间中，即 obs = [ωᵇ, gᵇ, cmd, q, q̇, aₜ₋₁, ζ]。这种设计在训练时需要外部提供 ζ 标签（通常来自 ground-truth 任务 ID），在部署时则由高层 skill selector 提供。然而，方案 α 将 ζ 完全暴露给策略，使得策略高度依赖该信号；一旦 ζ 评估出现偏差（如 selector 输出概率模糊），低层策略的行为可能退化。

    本文推荐方案 β（训练初期）与方案 γ（后期）组合的混合推断方案。方案 β 将 ζ 从观测中移除，低层策略的观测空间为 obs = [ωᵇ, gᵇ, cmd, q, q̇, aₜ₋₁]；同时，在 actor 网络中额外增加一个轻量化分支，以 ûₜ（推断的基座线速度）、v̂ₜ（推断的轮地距离）和 |cmd| 为输入，通过一个 2 层 MLP 输出 ζ̂ₜ = softmax(f(ûₜ, v̂ₜ, |cmd|))。该分支的输出在 actor forward 中与主策略分支分离（detach），不参与主策略的梯度反传，仅用于隐式推断当前技能标识。方案 β 的优点是：训练初期无须提供任何 ζ 监督信号，actor 自动从任务相关的速度-距离模式中聚类技能；在 ATEC 单环评测下，该方案表现出足够的鲁棒性。

    为加速收敛并避免方案 β 可能出现的单一技能坍塌（collapse），在训练后期引入方案 γ：添加一个交叉熵损失 L_ζ = CrossEntropy(ζ̂, ζ_gt)，用真实任务标识监督 ζ̂ 的输出。该损失与主策略的 PPO 损失联合优化，但梯度仅流入推断分支，主策略仍保持无偏。部署时切换到 ζ̂ 的最大概率值作为技能选择，无需外部信号。实际而言，MUJICA 的 S2 阶段（skill selector）本质上也走了一条类似路径——selector 输出概率分布并选择技能——只是其训练基于独立 reward 而非交叉熵监督。此处 β+γ 方案将推断与选择统一在低层 actor 内部，简化了层级。

    以下伪代码展示 actor forward 的两个分支（PyTorch 风格）：

    ```python class Actor(nn.Module): def __init__(self, obs_dim, hidden_dim, skill_dim): super().__init__() # 主策略分支：q(s) → mean, log_std self.policy_net = nn.Sequential( nn.Linear(obs_dim, hidden_dim), nn.ReLU(), nn.Linear(hidden_dim, hidden_dim), nn.ReLU(), nn.Linear(hidden_dim, action_dim) ) # 技能推断分支：g(û_t, v̂_t, |cmd|) → ζ̂ self.skill_encoder = nn.Sequential( nn.Linear(3 + 3 + 1, hidden_dim), # û_t (3), v̂_t (3), |cmd| (1) nn.ReLU(), nn.Linear(hidden_dim, hidden_dim), nn.ReLU(), nn.Linear(hidden_dim, skill_dim), nn.Softmax(dim=-1) )

    def forward(self, obs, est_vel, est_wheel_dist, cmd_norm): # 主策略 mean, log_std = self.policy_net(obs), torch.zeros_like(...) # 技能推断分支（detach） with torch.no_grad(): zeta_hat = self.skill_encoder(torch.cat([est_vel, est_wheel_dist, cmd_norm], dim=-1)) return mean, log_std, zeta_hat ```

    在实际实现中，主策略的隐层可与推断分支共享前几层，但梯度需隔离（如 stop gradient）。此设计在保留多技能统一控制能力的同时，避免了对显式 ζ 信号的依赖，提升了部署时的鲁棒性。

    04Reward Backbone: 加法 task + 乘法 envelope 的不可让步结构

    我们设计的 v2 奖励框架明确采用“加法 task + 乘法 envelope”的不可让步结构，其完整形式为：

    R = ( R_motion + α_en^{loco} R_en^{loco} + α_en^{arm} R_en^{arm} ) · exp( − R_aux − w_mc · C_DC )

    其中 R_motion 来自 2403.20001 的同款构造：R_motion = R_lin + α_ang R_ang，分别对应线速度跟踪与角速度跟踪的指数型奖励。R_en^{loco} 为移动通道的能量惩罚包络：R_en^{loco} = exp( − Σ_{i∈wheel∪leg} |τ_i||q̇_i| / (σ_en,x|v_x| + σ_en,z|ω_z|) )，分母使用基座线速度与角速度的加权和进行归一化，使能耗惩罚随运动强度自适应缩放。R_en^{arm} 为操作通道的能量惩罚：R_en^{arm} = exp( − Σ_{i∈arm} |τ_i||q̇_i| / σ_en,arm )，其分母固定，不按基座速度归一化，因为这会使机械臂的精细动作在低速运动时被过分压制。R_aux 收集辅助惩罚项：R_aux = w_col·C_collision + w_pose·R_pose + w_smooth·|Δa|²，其中 C_collision 为 MUJICA 原文定义的大腿/小腿碰撞计数，R_pose 为姿态奖励，|Δa|² 为动作平滑性惩罚，这是 MUJICA 与 2403.20001 的最小并集。C_DC 是 DC-motor 约束违反计数：C_DC = Σ_i 𝟙[|τ_i| ≥ τ^{i}_{limit}(q̇_i)] ，每个关节的力矩上限由官方电机手册给定的速度-力矩曲线决定（低速恒力矩、高速线性下降），是硬约束的软化版本。

    这一结构包含三个“不可让步”的设计要点，分别对应加法/乘法的角色与范围：

    (a) loco 与 arm 必须独立 R_en — 单一 α_en 的共享能量惩罚无法同时覆盖移动能耗（约 100 W）与操作能耗（约 10 W）。若合并，臂的细节动作在加法项中所占比重会被腿部的大幅度运动稀释，导致臂的执行质量退化。独立 R_en 使两条通道的能耗惩罚在加法主干中拥有独立的加权系数 α_en^{loco} 与 α_en^{arm}，各自调整幅度，互不干扰。

    (b) DC-motor 违反计数 C_DC 必须乘性进 envelope — C_DC 若以加法形式进入 R_aux，其效果会被 α_en 的调权稀释：当能量惩罚被系数降低时，DC 约束的违反代价也随之变小，失去对 sim-to-real 安全边界的强制保障。将其放入乘法指数包络，使其对整体奖励形成乘性衰减，无论加法项如何调整，任何 DC-motor 违反都会指数级降低总奖励，从而确保安全边界不可绕过。

    (c) 不引入 ζ 进 R_motion / R_en — 技能指示器 ζ 仅在观测空间中出现，而绝不进入奖励函数的任何项。若将 ζ 显式加入 R_motion 或 R_en，等于在奖励层面预先编码技能偏好（如“全向移动时关注角速度”、“爬高平台时忽略角速度”），这会破坏“涌现”哲学——技能应通过状态估计和自主探索自然分化，而非通过奖励模板预先定义。这恰恰避免了回到 MUJICA 原方案的显式 skill indicator 切换逻辑，使多技能整合在统一奖励框架下自主涌现。

    综上，v2 奖励通过加法主干保留任务导向的密集跟踪信号，再通过乘法包络引入物理安全约束与能量代价，且各自独立、不可合并或稀释，形成了 robust 且可扩展的奖励 backbone。

    06State Estimator (S1): 把 MUJICA 的 GRU online encoder 适配到 B2w

    在 MUJICA 原设计中，State Estimator（S1）采用 GRU 在线编码器，处理过去 H = 6 帧的本体感知序列，并预测四组中间量：̂v̂ ∈ ℝ³（基线 3D 速度）、û ∈ ℝ⁴（四轮各自的轮‑地距离）、ĉ ∈ ℝ¹⁸（各腿段的碰撞概率）、以及 ê ∈ ℝ¹⁶（隐状态 latent）。将其适配到 B2w 平台时，v̂ 与 û 直接等效借用（B2w 同样为四轮结构）；ĉ 则需扩展：MUJICA 原 18 维仅覆盖腿与轮段，B2w 额外配备 Piper 双臂，故将碰撞维度扩至 26——其中 18 维对应腿与轮段、8 维对应左/右臂各 4 段，即 ĉ ∈ ℝ²⁶。隐状态 ê 保持 16 维不变。训练损失沿用 MUJICA 的 Eq. 5–7：̂v̂ 与 û 采用 MSE 损失，ĉ 采用 BCE 损失，ê 则采用 SwAV 对比损失（由参考编码器对后续观测 oₜ₊₁ 编码得到监督目标 eₜ）。为何必须使用 State Estimator 而非直接馈入特权观测？关键约束在于 ATEC 评测场景中不提供 elevation map 或显式地形参数，仅有机载本体感知 + LiDAR + RGB‑D；actor 只能依赖本体感知到隐状态的映射来推断环境，这恰好是“盲走”（blind locomotion）范式的核心要求。训练阶段 S1 与 actor 策略共训（不冻结），原因在于隐状态 ê 的语义必须在 actor 动作空间中形成闭环，否则离线编码的 latent 无法匹配在线部署时的偏差——该论点将在下一章详细展开。

    07训练循环：单 PPO vs. MUJICA 两阶段的取舍

    MUJICA 原 paper 采用两阶段训练方案：S1 阶段联合训练 state estimator（GRU online encoder）、actor、reward critic 和 constraint critic，共 30 000 次迭代（在 IsaacLab 平台上运行 4 096 个并行环境，NVIDIA RTX 4090）；S2 阶段冻结 S1 的全部权重，仅训练 high‑level skill selector（skill indicator ζ 的显式分类器），额外 10 000 次迭代。该流程对 ATEC2026 B2wPiper 任务并非最优。原因有二：

    1. 部署约束：ATEC 的评测限时 30 分钟墙钟，policy.pt 需以单进程加载。MUJICA 的两阶段设计依赖在线 skill selector（S2）在推理时动态输出 ζ，这要求运行一个额外的轻量分类网络。墙钟限制下，单进程需同时承载 state estimator、actor 和 selector，selector 的网络开销虽小，但与其带来的冻结策略‑分布漂移隐患相比，得不偿失。

    2. 分布漂移下的 overfit：S2 阶段训练 selector 时，低层策略（actor、state estimator）的参数已冻结。当测试环境的地形、噪声分布与 S1 的训练分布存在漂移时，selector 只能基于固定策略空间的特征进行决策，容易 overfit 到 S1 数据分布下的局部最优选择。实验表明（Fig. 4 及 TABLE III），无 skill indicator 的方案往往在中等难度任务上成功率骤降，而冻结策略两阶段虽优于无 indicator 方案，但在高难度任务中（如高平台攀爬成功率低于 90%）仍不如端到端联合训练策略。

    因此，本模板推荐单 PPO 闭环训练方案，将总训练步数从 30 000 + 10 000 = 40 000 次迭代压缩为单阶段 40 000 次迭代。具体改动如下：

    - 将隐式 ζ̂ 分支作为 state estimator 的一部分，与 ê, v̂, û, ĉ 一同由 GRU online encoder 预测（参见 §6 State Estimator 设计）。ζ̂ 是长度为技能数的 softmax 概率向量，不单独作为图 2 中的独立模块。 - skill selector 不再作为 frozen‑policy gating（即不直接输出离散 ζ），而是作为 reward branch 的软加权：每个技能 ζ 对应一组权重系数 (α_en^{loco}, α_en^{arm}, w_mc)（参见 §4 Reward Backbone 的表达式 R = (R_motion + α_en^{loco} R_en^{loco} + α_en^{arm} R_en^{arm}) · exp(−R_aux − w_mc · C_DC)）。在训练时，由 ζ̂ softmax 概率对各组系数进行加权平均，使每批次数据隐式地、连续地受到多种技能导向的奖励项约束。 - 训练算法选用 P3O 的 penalty‑based constrained PPO（MUJICA Eq.8‑10 的框架），而非原版 PPO。原因：原版 PPO 的 clipping surrogate 会将 DC‑motor 约束惩罚项 C_DC 与优势函数估计混合，使约束信号被 reward advantage 的方差稀释；P3O 通过显式约束损失 L_C_i^CLIP + (J_C_i(π) - δ_i) 将约束的 penalty 系数作为可学习变量迭代更新，确保 C_DC 的梯度在 policy update 中产生稳定的安全驱动。

    权衡：该单 PPO 闭环方案失去 MUJICA 原设计的明确技能边界（即每个技能有独立的离散 ζ 指示）。在 MUJICA 中，skill indicator 作为观测输入显式告诉 actor“当前应执行哪种技能”，actor 可以针对性地激活相应 motor primitive；而在单 PPO 方案中，技能区分完全由奖励权重（α_en^{loco}, α_en^{arm}, w_mc 的加权均值）诱导，actor 需要从奖励信号中隐式学习不同地形对应的策略模式。这会降低单一技能的 peak performance（例如高平台攀爬的成功率可能下降），但获得了策略一体性——actor 的隐藏状态同时学习所有技能的共享表示，避免了 S2 阶段分布漂移下 selector 的 overfit。在 sim‑to‑real 部署中，策略一体性带来的鲁棒性（见 §D 实机实验中零样本迁移的成功）对有限 wall‑clock 的 ATEC 任务具有更高优先级。此外，DC‑motor 约束建模（§5）在单 PPO 中通过 P3O 的 penalty 机制全程参与梯度更新，进一步强化了长期可靠性。

    对于 ATEC TaskA（越野/地形穿越），我们将 §4 的奖励框架调整为：

    其中进度稀疏奖励每穿越 1 m 终点获得 +5（β_p = 5），该额外项不进入加法槽，避免与 R_en 的乘法调制产生耦合。α_en^{loco} 的初始值为 1.0；当训练地形难度增大时（如进入阶段 3 楼梯或阶段 4 综合地形），将其调低至 0.7，以放宽对能耗的惩罚，让机器人优先保证通过性。α_en^{arm} 固定为 0，因为 TaskA 要求“冻臂”，手臂完全不参与 R_en 计算。DC‑Motor 约束的权重 w_mc 设为高值 1.0：在越野地形上轮电机极易因频繁冲击和堵转而过载，硬约束护栏是防止损坏的关键。

    课程学习分为四个渐进阶段： - 阶段 1（2 k iter）：平地，前进速度 v_x 均匀采样于 [−1, 1] m/s。此时机器人只需基本轮式滚动，状态估计器给出的 û_t 通常较低（轮‑地距离小），ζ̂ 自然偏向 wheel‑roll‑mode，actor 保持低腿摆幅。 - 阶段 2（8 k iter）：浅斜坡（±15°），v_x ∈ [−2, 2] m/s。斜坡开始引入倾侧和抬腿需求，但 û_t 仍以低值为主，ζ̂ 仍以 wheel‑roll 为主，但部分上坡帧触发短暂 climb‑mode 倾向。 - 阶段 3（15 k iter）：楼梯（step ≤ 0.15 m），v_x ∈ [−2, 2] m/s。台阶迫使 û_t 周期性跳变，高 û_t 信号驱动 ζ̂ 偏 climb‑mode，actor 自动提高腿摆幅以跨越障碍。 - 阶段 4（15 k iter）：综合地形（坑、障碍、斜坡），v_x ∈ [−3, 3] m/s。多种混合地形要求 ζ̂ 在不同模式间实时切换：当 û_t 高（接近障碍或坑沿）时 actor 自动抬腿；当 û_t 低且 |v̂| 高（快速平路或下坡）时 actor 压缩腿摆幅、增大驱动效率。

    状态估计的角色在 TaskA 中被显式用于模式推断：û_t 高 → ζ̂ 偏 climb‑mode（提高腿摆幅）；û_t 低且 |v̂| 高 → ζ̂ 偏 wheel‑roll‑mode（降低腿摆幅、提升速度）。这一自适应机制使得单一策略从头到尾无需显式技能切换即可覆盖从平坦大道到复杂障碍的连续地形。

    09ATEC TaskB · 移动抓取：分层 vs. 一体策略

    对于 ATEC TaskB（拾取 + 投放），移动与操作必须在同一任务序列中共存，但接触不连续性（夹爪闭合/释放瞬间导致动力学突变）与连续移动的 RL 梯度优化存在根本冲突。我们比较两种实现方案。

    方案 X（分层）：底层分离为两个独立策略。底盘+腿策略采用 §4 的能耗式奖励结构（即 R = (R_motion + α_en^loco R_en^loco) · exp(-R_aux - w_mc C_DC)），专注地形穿越与定点停泊；臂+夹爪策略则使用密集 task reward（如末端执行器—目标位置距离、夹爪开合角度误差），并可借助 ACT 或行为克隆（BC）而非 PPO 来训练（参考 ATEC demo 的 `solution_act.py`）。高层 scheduler 以有限状态机形式在“移动至目标 → 停泊 → 抓取 → 移动至投放点 → 释放”共 5 个状态间切换，每个状态激活对应的低层策略及 skill indicator ζ_t。本模板的奖励设计仅覆盖底盘策略的 R_en^loco 部分；臂策略的奖励函数须另起一份独立训练脚本。

    方案 Y（一体）：单策略同时接收 R_en^loco 与 R_en^arm 的加权和，并将操作奖励（末端到目标距离、夹爪闭合状态）作为 R_motion 的扩展项纳入同一奖励流。但由于抓取动作的接触不连续性导致奖励曲面在抓取/释放瞬间出现尖峰，PPO 梯度的方差急剧增大，训练极易发散。

    推荐方案 X。理由是：抓取接触不连续与连续移动的 RL 梯度不兼容，分层训练不仅允许臂策略采用 ACT/BC 等更适合离散接触的控制方法，还能将底盘能耗奖励与操作密集奖励的 temporal scale 解耦。TaskB 评分对接采用三条稀疏奖励：抓取成功 +10，投放成功 +10，任务完成时间余量 +5（即若 20 s 内完成触发额外奖励）。这三项 bonus 直接加在 episode return 尾部，不进入 \(R_{\text{en}}\) 的分母，从而不影响奖励归一化的稳定性。

    10ATEC TaskD · 越障+环境改造：能耗解耦的临界点

    在 ATEC Task D（推/拉/堆叠物体，“改造动作”）中，机器人的臂电机需输出高 torque 与高 velocity，其瞬态功率 |τ_arm|·|q̇_arm| 远大于普通移动时的关节能耗。若按 §4 的通用奖励结构将改造动作的能量消耗直接计入 R_en^{loco}（即采用按 |v_x| 归一化的 R_en^{loco}），策略极易收敛到退化模式——“最省能 = 不动臂”，从而完全放弃改造任务。为此，我们在 Task D 中执行关键的解耦策略：

    (1) R_en^{arm} 的分母改用固定常数 σ_en,arm（不再以 |v_x| 归一化），使臂能耗的 penalty 幅度与移动速度解耦，避免高速奔跑时被动放大臂能耗惩罚；(2) α_en^{arm} 在 Task D 中设为 0.2，该值极小，相当于几乎不约束臂的能耗，使策略自由探索高功率动作；(3) 改造任务成功（物体抵达目标位置）的奖励以 sparse +50 直接加入 task reward (R_motion)，彻底绕过 R_en 通道，确保改造动机不受能耗惩罚衰减；(4) C_DC（DC-motor 硬约束，§5）在 Task D 同样保持激活：臂电机一旦超出速度‑扭矩包络即触发过载保护，导致整机关机、损失分数。因此，臂的能耗不被软惩罚限制，但物理极限仍由硬约束兜底。

    这一设计的本质是将“能效”与“任务动机”解耦：臂的动作受 C_DC 物理上限保护，而不受 R_en 的经济性引导。相比之下，纯 energy paper（仅用总能耗惩罚）在 Task D 中会因为单值化能耗而陷入不动臂的局部最优——这正是 MUJICA 多约束思路（分工明确的奖励项 + 硬约束 + sparse task reward）真正胜出的场景。结合 §7 的两阶段训练，该解耦结构使得策略在推/拉/堆叠任务中仍能充分挖掘臂的能力，同时不会牺牲运动安全。

    为确保 ATEC demo 的 inference 接口与训练代码严格一致，本节将 obs/action 的维度定义、拼接顺序及接口契约进行精确锁定，并给出可直接嵌入 `server.py` 的校验片段与端到端测试 checklist。

    13Sim-to-Real 与部署风险

    在 MUJICA 框架向 B2w 轮腿机器人迁移的 sim-to-real 过程中，需系统应对以下已知风险。风险 1：能耗系数 σ_en 的 sim-to-real gap。仿真中地面摩擦固定为 0.7，而实物地面摩擦在 0.4–0.9 范围内波动，导致电机实际功耗偏离仿真。为此，部署后前 1 分钟进行 warmup 阶段，通过采集真实功耗数据，利用在线 PID 控制器实时校准 σ_en，使能耗估计回归合理区间。风险 2：DC 电机 τ_limit 在实物上因温度漂移而下降。MUJICA 原框架虽已将硬约束建模为 τ_limit(˙q)，但温度效应未体现。部署时直接采用 τ_limit_deploy = 0.9 × τ_limit_sim，保留一定安全 margin，防止过热触发电机保护。风险 3：高层技能选择器 ζ̂ 在 ATEC 评测环境下可能收敛到单一技能（技能 collapse），丧失多技能切换能力。这与 MUJICA 原设计中 skill indicator 的稀疏监督有关。作为 fallback，保留额外训练轨迹中的 ζ supervision（对应原有 γ 方案），必要时回退以维持选择器多样性。风险 4：单 PPO 作为基线收敛速率显著低于 MUJICA 的两阶段训练（S1: 30 000 次迭代、4 096 环境并行；S2: 10 000 次迭代）。实际部署时需预留 2× wall-clock 时间预算，确保策略达到同等性能。风险 5：Piper 臂的扭矩反馈精度低（无电流传感），无法直接获得真实 τ。采用基于位置/速度的估计扭矩 τ̂ = K_p·(q−a) + K_d·q̇ 替代，误差在一定范围内可接受，对抓取类任务影响可控。按缓解紧急度排序：风险 3（技能 collapse 导致任务失败） > 风险 2（电机过热可能损坏硬件） > 风险 1（能耗偏差影响续航） > 风险 4（训练时间可调度） > 风险 5（力矩估计误差可被奖励设计吸收）。这些措施共同保障了 MUJICA 框架在 B2w 实物上的安全、可靠迁移。

    14实施 Roadmap 与提交清单

    实施 Roadmap 与提交清单采用 10 周 wall‑clock，每一步聚焦特定模块集成与验证。 W1：从 ATEC‑Isaac‑Velocity 派生 B2wPiper 环境，集成 24 维 action 解码（12 个关节角偏移 + 4 个轮速 + 8 个其他自由度）及 DC 约束计算（速度‑依赖转矩限值 + 小腿关节位置‑依赖余弦限值，参见 Fig. 3）。 W2：实现 state estimator（GRU 处理 H=6 帧，输出 ̂v̂∈ℝ³、û∈ℝ⁴、ĉ∈ℝ¹⁸、ê∈ℝ¹⁶）、actor、双 reward critic 与 constraint critic（均基于 P3O 框架），形成单 PPO 闭环。privileged observation 仅用于 critic。 W3：Task A 课程训练 40k iter（比 MUJICA 原论文的 30k 多 10k 以适配更高难度地形）。对照 baseline（无 R_en、无 C_DC）验证 CoT 与跟踪误差；C_DC 的约束阈值 δ_g 设为 12，违反率预期如图 5 从 >90% 大幅降低。 W4：Task A 部署测试 + ζ̂（skill indicator）推断质量检查，通过混淆矩阵评估技能选择器在困难斜坡与楼梯上的区分精度。 W5–6：扩展至 Task B（分层，含夹爪动作切换）与 Task D（解耦 R_en，对推/堆叠任务单独施加能耗惩罚项）。 W7：完成消融 18 组（包括移除 velocity 估计、wheel‑ground 距离、collision 估计、C_DC、R_en 及不同课程策略），整理结果 → 选最优配置（预期为完整 MUJICA + C_DC + 课程学习）。 W8：部署整合，输出 solution.py（含 state estimator + actor + selector）与 policy.pt；在模拟中执行 3 次 30 分钟连续评测（混合 terrain 和 command）。 W9–10：提交准备：奖励函数 reward.py、配置 config.yaml（含 30k S1 + 10k S2 迭代，50 Hz 控制频率）、需求 requirements.txt、模型 policy.pt，以及 runs/Task_A.csv 等消融数据。

    提交包结构： ``` solution/ policy.pt reward.py config.yaml requirements.txt runs/ Task_A/ Task_B/ Task_D/ ablation.csv (包含各配置参数及 Wrec, η, success rate 等) ``` 所有步骤中，DC 约束建模（速度‑转矩限值 + 小腿余弦依存）与 skill indicator 的显式馈入（ζ_t ∈ {0,1,2}）为关键创新点，需在 config.yaml 中对应标注。消融实验需复现原文 Fig.4 的多难度比较逻辑，确保 baseline 覆盖 w/o Velocity、w/o Wheel Height、w/o Collision 及 Vanilla PPO。

    15Conclusion: 两篇论文的真正贡献是『可解释的奖励工程』

    MUJICA 的真实贡献不在“单网络统一多种技能”这一表层，而在于用一套可解释的奖励工程使“能耗驱动的步态涌现”与“多技能统一策略”这对看似矛盾的追求共存。具体而言，三层设计共同支撑了这一论点。

    （1）哲学：能耗驱动涌现 + 多技能统一不冲突，前提是 ζ 作为推断状态而非观测输入。 MUJICA 将技能指示器 ζ 设计为隐变量——由状态估计器从本体感知推断得到，而非直接作为观测馈入策略。这保留了端到端强化学习的涌现空间：机器人不被强制“看到”当前是哪个技能在执行，而是通过自身状态（基座速度、轮地距离、碰撞）自然切换行为模式。图4 展示，配备完整状态估计的 MUJICA 在楼梯、斜坡、楼梯恢复三个任务上都持续优于无指示器的变体，印证了 ζ 不破坏涌现。

    （2）结构：单 PPO 闭环 + 加法 task / 乘法 envelope 骨架 + 运动/操作能耗解耦的双 R_en 设计，是承担 ATEC 三任务的最小一致框架。 MUJICA 的低层策略采用单 PPO + 乘法约束（P3O）结构：主奖励 R_cmd 为速度跟踪（加法项），DC 电机约束以乘法 penalty 形式嵌入 envelope，同时通过 skill indicator 对不同任务分别施加组合权重。这种结构天然支持运动（R_en^{loco}）与操作（R_en^{arm}）的能耗解耦——仅需在后续扩展中为臂部引入独立的 R_en 项，即可在不改动主体框架的前提下覆盖 ATEC 的移动抓取与越障改造任务。表 III 中，无指示器的统一奖励方案直接崩溃，而 MUJICA 在三任务连续执行中达到约 95%、95%、91% 的成功率，说明该结构是“多技能集成”的最低必要工程范式。

    （3）工程：四个可直接写入训练脚本的落地点。 - DC 电机软约束已嵌入 reward envelope：每条关节的 velocity-dependent torque limit 由电机手册数据建模（图3），直接作为安全护栏。 - α_en 分组：不同技能下的约束松弛系数可分别设置（如高平台攀爬时允许大腿短暂超限），文中通过 skill indicator 间接实现。 - σ_en B2w 缩放：约束代价的缩放因子随地形难度递进，仅在课程学习中提及，但可显式参数化。 - 状态估计扩展：MUJICA 已实现 base velocity、wheel-ground distance、segments collision 三个估计头；扩展到臂段碰撞只需在观测中加入臂的位置/速度特征，网络结构无需改变。

    本框架相对原版 energy template 的边际增益主要体现在： - 多技能能力（继承自 MUJICA），但保留涌现哲学（ζ 为隐变量，不硬编码行为）。 - DC 电机 sim-to-real 安全护栏：硬约束建模使 95% 的仿真样本位于可行域（图5），无约束时超 90% 跳出安全区。 - 移动/操作能耗解耦：这是 ATEC Task B/D 真正翻车点的解药——在连续序列中，臂部分离的 R_en 项可单独调节，避免移动策略被高 torque 操作干扰。

    下一步建议：先在 Task A（越野穿越）上运行 baseline + v2 对比，确认 ζ 隐式推断不破坏涌现，再扩展至 Task B/D。
