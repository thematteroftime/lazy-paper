# v1.12 Phase 4 — Two-Stage Prompt Tailoring Design Spec

> **One-line:** Add a cheap pre-stage LLM call that reads the paper's title + abstract + first chapter, produces a per-paper "prompt augmentation block" (domain framing, terminology, metric patterns, comparator style), and prepends it to s08's `_STRUCTURED_SYSTEM` so the thinking model gets a paper-tailored prompt instead of a one-size-fits-all template stuffed with hypothetical examples.
>
> **Why now:** Phase 3 proved that **example-stuffing for generalization regresses quality** (meng2024 -9.4pp, ali2025 -4.4pp when ML examples were added to the prompt). The right abstraction is structural: keep the system prompt as a generic *template*, and let a small LLM tailor it per-paper at run time.
>
> **What was already reverted (cleared the runway):** Phase 3b (prompt compression) and Phase 3c (example-stuffing generalization). Phase 3a (Chinese mirror) is preserved — it's docs-only.

---

## 1. Goal

Two-stage prompt construction for s08 compose, gated by
`LAZY_PAPER_PROMPT_TAILOR=1` (default OFF until measured).

**Ship gate (per paper):**
- `faithfulness ≥ Phase 2 baseline ±1pp` on **both** meng2024 and ali2025_flash
- AND: at least one paper gains ≥+2pp

If neither: stays opt-in or reverted; lessons recorded for Phase 5.

---

## 2. Background — what Phase 3 taught us

Cumulative RAGAS faithfulness for meng2024 across phases:

```
v1.11.5 baseline      0.6435
Phase 2 anchored      0.5447   (apparent −9pp; was the empty-quote bypass closing)
Phase 3 (full)        0.4510   (additional −9pp; mostly prompt compression + examples)
Phase 3 (a+c, no b)   0.5008   (some recovery; 3c examples still hurt)
```

Empirical finding: **adding example clauses to the system prompt** (Phase 3c's "Smith et al. ResNet-50" lines) **degrades behavior on materials papers**. The LLM treats the extra examples as "permission to write differently," even though the examples were meant to be domain-illustrative.

The user's framing of the fix:
> 通用化完全不需要给例子，给的是模板，给的是架构，给的是方法论。
> 开始时用最简单最轻量的模型扫一遍数据得到大概领域和范围，
> 有了基本的信息后，比如还可以和摘要结合起来，
> 让小模型做一个简单的总结并且和通用模板结合起来得到更有针对性的prompt，
> 之后这个新的prompt才是思考模型能够使用的。

Translation: generalization is structural (templates / architecture / methodology),
not example-stuffing. The right pattern is a pre-stage that reads the paper,
synthesizes per-paper context, and emits a focused prompt the thinking
model then uses.

## 3. Design

### 3.1 New artifact: `s06_context/prompt_augment.yaml`

A YAML written by a new sub-step at the end of `s06_context.run()`. Schema:

```yaml
# Generated by s06_context prompt_tailor sub-step.
# Consumed by s08 compose when LAZY_PAPER_PROMPT_TAILOR=1.

domain_framing: |
  This paper investigates lead-free relaxor antiferroelectric ceramics
  for energy storage. Methods include ceramic synthesis (sol-gel /
  solid-state reaction), P-E loop measurement, SEM/TEM microscopy, and
  defect-dipole engineering. The dominant evaluation framework is
  recoverable energy density (W_rec) and efficiency (η) at applied
  electric field (kV/cm).
terminology:
  - {term: "W_rec", note: "recoverable energy density, J/cm³; preserve subscript"}
  - {term: "η", note: "efficiency, %; Unicode eta required"}
  - {term: "E_b", note: "breakdown field, kV/cm"}
  - {term: "PNR", note: "polar nanoregion (preserve abbreviation)"}
  - {term: "NBT-BMZ", note: "the system being studied; chemical formula"}
metric_patterns:
  - {kind: "energy_density", regex: "\\d+\\.?\\d*\\s*J/cm[³3]"}
  - {kind: "efficiency", regex: "\\d+\\.?\\d*\\s*%"}
  - {kind: "field", regex: "\\d+\\s*kV/cm"}
comparator_style:
  format: "<Author> et al. (year) reported <metric>=<value> in <system>"
  example_from_paper: "Jiang et al. reported W_rec=2.94 J/cm³ in Ca²⁺/Nb⁵⁺-codoped Bi₀.₅Na₀.₅TiO₃"
generated_by: prompt_tailor_v1
generated_at: <ISO timestamp>
```

For an ML paper, the same schema would auto-fill:

```yaml
domain_framing: |
  This paper proposes a text-to-image diffusion prior. Methods include
  transformer training, CLIP-based contrastive learning, and FID-based
  evaluation against MS-COCO. The dominant evaluation framework is
  zero-shot FID and human-judged photorealism / diversity.
terminology:
  - {term: "unCLIP", note: "the proposed method; preserve verbatim"}
  - {term: "FID", note: "Frechet Inception Distance; preserve abbreviation"}
  - {term: "CLIP", note: "Contrastive Language-Image Pretraining"}
metric_patterns:
  - {kind: "fid_score", regex: "FID\\s*[=:]?\\s*\\d+\\.?\\d*"}
  - {kind: "percentage", regex: "\\d+\\.?\\d*\\s*%"}
comparator_style:
  format: "<Author> et al. (year) achieved <metric>=<value> with <method>"
  example_from_paper: "Ramesh et al. reported FID=10.39 with autoregressive prior"
```

### 3.2 New prompt: `llm/prompts/prompt_tailor.md`

The pre-stage LLM reads:
- `context.yaml` already-extracted: title / system / abbreviations / keywords / headline_metrics
- `s03_chapter/chapters/chapter_001_INTRODUCTION.md` first 3000 chars

Outputs JSON matching the schema above. Instructions emphasize:
- Extract terminology FROM THE PAPER, don't invent
- metric_patterns regex should match patterns actually appearing in the paper text
- comparator_style example must be a real instance from the paper

### 3.3 s06 runner integration

End of `stages/s06_context/runner.py::run()`:

```python
if os.environ.get("LAZY_PAPER_PROMPT_TAILOR", "0") == "1":
    from stages.s06_context.prompt_tailor import generate_prompt_augment
    try:
        augment = generate_prompt_augment(context=data, chapters_dir=chapters_dir)
        dump_yaml(out_dir / "prompt_augment.yaml", augment)
    except Exception as exc:
        (out_dir / "prompt_tailor.failed").write_text(repr(exc))
```

Soft-degrade on any failure (file marker only; s08 falls back to vanilla prompt).

### 3.4 s08 consumption

In `stages/s08_section_compose/structured.py::compose_structured`:

```python
augment_path = context_dir / "prompt_augment.yaml"
augment_block = ""
if augment_path.exists():
    aug = load_yaml(augment_path)
    augment_block = _render_augment_block(aug)  # ~10 LOC inline render

system_prompt = (augment_block + "\n\n" + _STRUCTURED_SYSTEM) if augment_block else _STRUCTURED_SYSTEM
```

`_render_augment_block(aug)` produces a focused 5-15 line block:

```
## This paper — domain context

<domain_framing>

## Terminology to preserve verbatim

- W_rec: recoverable energy density, J/cm³; preserve subscript
- η: efficiency, %; Unicode eta required
- ...

## Comparator citation style

Format: <Author> et al. (year) reported <metric>=<value> in <system>
Example from this paper: "Jiang et al. reported W_rec=2.94 J/cm³ in ..."
```

No examples from OTHER papers. The augment block is the per-paper
specialization; `_STRUCTURED_SYSTEM` remains the generic methodology
template.

### 3.5 Why this beats Phase 3c

| Aspect | Phase 3c (example-stuffing) | Phase 4 (prompt tailoring) |
|---|---|---|
| Generalization mechanism | Static prompt with ML+materials examples | Per-paper auto-tailored prompt |
| meng2024 behavior | LLM distracted by ML examples → −4.4pp | LLM sees only meng2024 terminology |
| ali2025 behavior | Same distraction | LLM sees only PZO/FHC terminology |
| ML-paper behavior | Has ML examples (good) but stuck with materials too (bad) | Gets ML-specific framing (clean) |
| Cost | Same | +1 cheap LLM call per paper (~$0.001) |
| Maintenance | Hand-curate examples per domain | Zero — pre-stage discovers domain |

## 7. Open questions

1. **Default ON or OFF?** Spec says OFF until measured (gate must pass).
   Once gate passes, propose default ON in a follow-up commit. **DEFAULT
   to OFF for Phase 4 ship.**

2. **Does `compose_structured` also need a code path to RE-tailor on retry?**
   No — augment is per-paper, not per-section. Tailor once, use everywhere.

3. **What if a paper has no abstract section?** Pre-stage LLM gets only
   `context.yaml` + first 3000 chars of intro. If intro is empty too, the
   pre-stage soft-degrades and s08 falls back to vanilla prompt.

---

## 8. Ship rationale

Phase 4 succeeds where Phase 3 failed because the **generalization
substrate is the architecture, not the prompt body**. The system prompt
stays clean, focused, and provably effective on materials science (where
v1.11.5 already worked). Cross-domain capability comes from the
pre-stage's ability to read any paper and emit appropriate framing — no
need to anticipate every domain at prompt-authoring time.

If the gate passes on the 2 demo papers, future cross-domain validation
(e.g., adding hif_2 / hif_1 golden_qa) becomes a measurement task, not
an engineering task.
