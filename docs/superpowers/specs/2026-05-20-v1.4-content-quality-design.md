# lazy-paper v1.4 — Content Quality Design Spec

> **Status**: design-approved, awaiting writing-plans handoff. Brainstormed and library-validated 2026-05-20.
>
> **Tagline**: lazy-paper v1.4 becomes a *single-paper deep-research agent*. Built on **5 mature OSS libraries** (llama-index, bm25s, instructor, pydantic-ai-slim, OpenAI SDK) plus ~185 LOC of project-specific glue + ~516 LOC vendored from Onyx (MIT). No reinventing wheels. Reference family: Onyx / AI-Scientist-v2 / DeepScientist / claude-code — patterns adopted via libraries that already implement them, not re-implemented from scratch.

## 1. Goal

Close three content-quality defects observed in v1.3.3 (per `docs/v1_4_roadmap.md`):

1. **Template-driven hallucination** — yang2025 ch01 fabricated "8.6 J/cm³ at η=85%" in a paper with no energy-storage numbers.
2. **Quoted-symbol drift** — meng2024 conflated E_b=348 kV/cm with test-field=340 kV/cm.
3. **Missed direct source facts** — meng2024 ch10 said "synthesis not addressed" while source explicitly stated "tape-casting".

Each defect maps to a different root cause: no typed entity grounding (1), no post-generation source check (2), keyword-matched excerpts missing relevant text (3). Closing all three requires the architecture below.

## 2. Mental model

```
PaperDB (built once per paper; immutable artifact, mostly library-built)
  ├─ paper_kg.parquet           closed 10-type entity/relation graph
  │                             (custom Pydantic schema, instructor extracts)
  └─ retrieval.parquet          parent-child chunks + dense vec + BM25
                                (llama-index chunks + bm25s sparse + numpy dense)

Section composer (15 instances per paper, tool-using agents)
  ├─ pydantic-ai-slim Agent     (loop infrastructure done by library)
  ├─ Tool: query_kg              custom 10-line tool body
  ├─ Tool: retrieve              custom 8-line tool body
  ├─ Tool: check_source          custom 8-line tool body
  └─ Tool: emit_section          custom 4-line tool body

Reviewer (runs after composer emits)
  ├─ Tier 1: Python regex (free, deterministic)
  │     - numeric / Fig.N / chem-formula → grep source via KG
  │     - unit normalization (custom _units.py, ~50 LOC)
  └─ Tier 2: instructor + Pydantic Flag class
        - JSON-mode structured critique with retry/parse
        - dimensional scoring: quote-fidelity / grounding / synthesis-depth 1-4
        - one revision round max
```

**Architectural principle: use libraries where they fit; write only project-specific glue.**

## 4.1 Vendored code from MIT-licensed OSS

Onyx (formerly Danswer) is a monorepo full-stack app, **not pip-installable**. But one of its files is high-quality, MIT-licensed, and low-coupling enough to vendor:

| Source | Target in lazy-paper | LOC | Adaptations |
|---|---|---|---|
| `onyx-dot-app/onyx:backend/onyx/chat/citation_processor.py` | `llm/citation/stream_processor.py` | ~516 | (1) Preserve MIT header + `Copyright (c) Onyx (formerly Danswer)` attribution. (2) Replace 3 Onyx imports (`SearchDoc`, `CitationInfo`, `STOP_STREAM_PAT`) with local Pydantic models we define alongside. (3) No logic changes — keep the regex state machine and three rendering modes (HYPERLINK / KEEP / REMOVE) verbatim. |

**Why vendor instead of dep**: Onyx isn't on PyPI; the PyPI `onyx` package is unrelated (squatted, broken metadata). The real `onyx-dot-app/onyx` repo declares `version="0.0.0"` and is Docker-deployed.

**Why vendor instead of reimplement**: hand-rolling streaming citation handling (`[1]` / `[1,2]` / `[[1]]` / code-block escaping / hyperlink mode toggle) is ~200 LOC of subtle regex easy to get wrong. Onyx's version is battle-tested in production.

**License compliance**: third-party-attribution file at `THIRD_PARTY_NOTICES.md` (new) lists the source repo, commit SHA, MIT license text, and modifications applied. Header inside the lifted file preserves original copyright.

Rejected from lifting (also from Onyx):
- `dr_loop.py` — couples to Postgres `get_session_with_current_tenant` + chat-state primitives we don't need.
- `chunker.py` — llama-index's `SentenceSplitter` is sufficient.
- `search_nlp_models.py` — couples to a separate `model_server` HTTP service; `litellm.rerank()` direct call is simpler if we ever need reranking.

## 8. Tool contracts (v1.4.0 — pydantic-ai signatures)

Each tool is a `@agent.tool async def` with Pydantic-typed args. pydantic-ai handles schema generation, validation, and the dispatch loop.

```python
@agent.tool
async def query_kg(ctx: RunContext[SectionContext], entity_type: str,
                   filter: dict | None = None) -> list[dict]:
    """Return entities of the given type, optionally filtered.
    Types: material / dopant / parameter / value / unit / figure / table /
           claim / method / comparator."""
    return ctx.deps.kg.query(entity_type=entity_type, filter=filter)

@agent.tool
async def retrieve(ctx, query: str, top_k: int = 8,
                   entity_boost: list[str] | None = None) -> list[dict]:
    """Return text chunks ranked by hybrid dense + BM25 with optional
    entity-id boost. top_k clamped to 12."""
    chunks = ctx.deps.retriever.retrieve(query, top_k=min(top_k, 12),
                                         entity_boost=entity_boost or [])
    return [c.to_dict() for c in chunks]

@agent.tool
async def check_source(ctx, claim: str, expected_value: str | None = None) -> dict:
    """Substring + unit-normalized search of source for the claim.
    Returns {found, span: (doc, start, end), evidence: str}."""
    return ctx.deps.retriever.check_claim(claim, expected_value)

@agent.tool
async def emit_section(ctx, draft: str) -> str:
    """Terminal call. Returns draft. pydantic-ai's tool-loop ends after this."""
    if not _has_min_citations(draft, n=1):
        # encourage agent to retrieve more before emitting
        raise ToolValidationError("draft needs ≥1 [span:...] citation marker")
    return draft
```

Agent loop is fully managed by pydantic-ai. We only write tool bodies (~30 LOC total) and the system prompt.

## 9. Reviewer contract

```python
# Regex tier (v1.3.4) — pure Python, no LLM
class Flag(BaseModel):
    span: tuple[int, int]
    claim: str
    problem: Literal["numeric_not_in_source", "fig_not_in_yaml",
                     "formula_not_in_kg", "unit_mismatch"]
    evidence: str | None = None

def regex_check(draft, source_docs, kg, fig_yaml) -> list[Flag]: ...  # ~40 LOC

# LLM tier (v1.4.0) — instructor + Pydantic
class CritiqueRevision(BaseModel):
    revised_draft: str
    quote_fidelity: int = Field(ge=1, le=4)
    grounding: int = Field(ge=1, le=4)
    synthesis_depth: int = Field(ge=1, le=4)
    notes: str

def llm_review(draft, flags, evidence) -> CritiqueRevision: ...   # ~15 LOC
```

instructor handles JSON-mode wire format, retry, and validation against the Pydantic schema. We just call it.

## 10. Error handling

| Surface | Failure | Behavior |
|---|---|---|
| KG extract | instructor parse fails twice | Marker `kg_extract.failed`; s08 uses keyword fallback; pipeline completes |
| Retrieval build | embedding API down | Marker `retrieval.failed`; s08 uses keyword fallback |
| pydantic-ai agent | iter cap hit | Library raises `UnexpectedEndOfStream` or similar; we catch and force `emit_section` from last partial draft |
| pydantic-ai agent | LLM error mid-loop | Library auto-retries once via `ModelRetry`; if still fails, fall back to legacy compose for that section only |
| Reviewer regex | fatal Python (rare) | Skip, accept draft, log |
| Reviewer LLM (instructor) | parse fail after retry | Accept original draft, write unresolved flags |

Universal rule: **soft-degrade per-paper, per-section. Never abort the pipeline.**

## 13. Roll-back strategy

- Each release is its own git tag (`v1.3.4`, `v1.4.0`).
- PaperDB files (`paper_kg.parquet`, `retrieval.parquet`) are per-run-directory; deleting them returns s08 to keyword-fallback mode on next run.
- Hard rollback: `git checkout v1.3.3` returns to known-good pre-v1.4 state.
- Soft rollback: keep code on v1.4.0 but delete PaperDB files for affected papers; runs auto-degrade to v1.3.3-equivalent behavior.

## 14. Deferred to v1.5

- **Findings Memory CONSUMER** (DeepScientist pattern): cross-section claim accumulator that informs next section composition. v1.4 writes `findings.yaml`; v1.5 reads it. Defer because cross-section state is premature optimization before the basic agent is validated.
- **Multi-reviewer ensemble**: 2-3 reviewer personas vote on each section. Premature until single-reviewer regression is understood.
- **DSPy Signature scaffold** for prompts: maintenance win, not quality win. Defer to v2.0.

## 16. What this spec is NOT

- **Not a re-architecture of the 9-stage pipeline.** STAGE_ORDER stays 9. s06 gets KG as a sub-step; s08 gets agent + reviewer. No new stages.
- **Not a wheel-reinvention exercise.** We use llama-index for chunking/index, bm25s for sparse, instructor for typed outputs, pydantic-ai for agent loops. Only ~185 LOC of project-specific glue + ~516 LOC vendored from Onyx (MIT) remains.
- **Not a quality miracle.** We expect to close the three named defects; we do **not** claim to eliminate all hallucination or guarantee SOTA synthesis.
- **Not a license risk.** All 5 direct deps are MIT or Apache-2.0. AI-Scientist-v2 code (RAIL) is not lifted.
